# Архитектура olcrtc-core

## Что это и зачем

`olcrtc-core` — сетевое ядро, которое позволяет туннелировать произвольный TCP-трафик через WebRTC-сессии легальных видеоконференц-платформ. Идея: эти платформы, как правило, не блокируются даже в условиях жёсткой фильтрации, поскольку используются для рабочих звонков. `olcrtc-core` использует их инфраструктуру, отправляя данные внутри WebRTC DataChannel или видеопотока.

Ядро написано на Go и спроектировано для использования в высоконагруженных сценариях: минимальные аллокации, строгое управление жизненным циклом горутин через `context.Context`, отсутствие глобального состояния (кроме реестров провайдеров и транспортов).

## Ключевые архитектурные решения

### 1. Реестр компонентов

Каждый провайдер и транспорт регистрируется через типобезопасный `registry.Registry[T]`. Это позволяет:
- добавлять новые провайдеры/транспорты без изменения ядра;
- проверять конфигурацию на этапе инициализации, а не в runtime;
- легко писать тесты с мок-провайдерами.

```go
// Провайдер регистрируется один раз при инициализации пакета:
provider.Register("telemost", func(cfg provider.Config) (provider.Provider, error) {
    return telemost.New(cfg)
})
```

### 2. Управление жизненным циклом

Все долгоживущие компоненты реализуют интерфейс `lifecycle.Component`:

```go
type Component interface {
    Start(ctx context.Context) error
    Stop(ctx context.Context) error
    State() lifecycle.State
}
```

Встраиваемый `lifecycle.Base` обеспечивает:
- атомарные переходы между состояниями (`Idle → Running → Stopped`);
- невозможность перезапуска остановленного компонента;
- корректный `graceful shutdown` с таймаутом через переданный `context`.

### 3. Транспортный уровень

Все транспорты реализуют `transport.Transport`, расширяющий `net.Conn`. Это означает, что слой мультиплексирования (smux), шифрования и диспетчеризации не зависит от конкретного транспорта — их можно менять в конфигурации.
Session.run() └─▶ provider.Join() ← Соединение WebRTC-пиром └─▶ transport.Dial() ← DataChannel / VideoChannel / SEI └─▶ session.ClientHandshake() ← X25519 DH + HKDF └─▶ newEncryptedConn() ← XChaCha20-Poly1305 AEAD └─▶ mux.NewClientSession() ← smux multiplexer └─▶ socks5.Handler.Serve() ← Входящий SOCKS5

### 4. Криптографическая модель

Подробно описана в [docs/security.md](security.md). Кратко:

1. Обе стороны имеют общий 256-битный pre-shared key (PSK).
2. При установке соединения каждая сторона генерирует эфемерную пару X25519.
3. Выполняется ECDH. Shared secret + PSK + публичные ключи обеих сторон подаются в HKDF-SHA256, из которого выводятся два независимых сессионных ключа (send/recv).
4. Все данные зашифрованы XChaCha20-Poly1305 с рандомным nonce на каждое сообщение.

Это обеспечивает **forward secrecy**: компрометация PSK не раскрывает прошлые сессии.

### 5. Диспетчер и маршрутизатор

`dispatcher.Dispatcher` принимает входящие mux-потоки на стороне сервера и на основе метаданных потока (адрес назначения, записанный клиентом в начало потока) вызывает нужный `Handler`. По умолчанию есть один хендлер — TCP-форвардер.

`routing.StaticRouter` оценивает список `Rule` по порядку и возвращает первое совпадение. Расширяется реализацией интерфейса `routing.Router`.

## Сравнение с xray-core

| Аспект | xray-core | olcrtc-core |
|--------|-----------|-------------|
| Протоколы | VMess, VLESS, Trojan, SS, … | Только olcrtc |
| Транспорты | WebSocket, gRPC, mKCP, … | DataChannel, Video, SEI, VP8 |
| Инфраструктура | Собственные серверы | WebRTC-комнаты провайдеров |
| Обфускация | TLS fingerprinting | Легитимный WebRTC-трафик |
| Криптография | Зависит от протокола | X25519 + XChaCha20-Poly1305 + HKDF |
| Маршрутизация | GeoIP, domain rules | Metadata rules (расширяемые) |

## Производительность

- **Буферный пул** (`common.BufPool` / `sync.Pool`) исключает аллокацию на горячем пути чтения/записи.
- **Атомарные счётчики** (`sync/atomic`) вместо мьютексов для метрик.
- **smux v2** с окном 4 МиБ обеспечивает эффективный конкурентный мультиплексинг.
- **Zap** как логгер не аллоцирует при уровне выше текущего.
