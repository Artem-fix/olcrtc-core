# Безопасность

## Криптографическая модель

### Pre-Shared Key (PSK)
```
Оба участника заранее договариваются об общем секрете — 256-битном ключе. Ключ **никогда не передаётся по сети** и используется только как входной материал для HKDF.
PSK (256 бит) ──┐ ├──▶ HKDF-SHA256 ──▶ SessionKey_Send (256 бит) Общий секрет ──┘ └──▶ SessionKey_Recv (256 бит) (X25519 DH)
```
### Схема установки сессии (Handshake)
```
Клиент Сервер │ │ │── ClientPubKey (32 байта) ───────────▶│ │ │ Генерирует ServerKeypair │◀──── ServerPubKey (32 байта) ────────│ │◀──── AEAD{"olcrtc-hello"} ────────────│ │ │ │ DH(ClientPriv, ServerPub) │ DH(ServerPriv, ClientPub) │ HKDF(PSK, SharedSecret, │ HKDF(PSK, SharedSecret, │ ClientPub, ServerPub) │ ClientPub, ServerPub) │ → SendKey, RecvKey │ → RecvKey, SendKey │ │ │── AEAD{"olcrtc-ready"} ─────────────▶│ │ │ ├── Зашифрованный канал готов ──────────┤
```
### Алгоритмы

| Примитив | Алгоритм | Параметры |
|----------|----------|-----------|
| ECDH | X25519 | 256-битный ключ |
| KDF | HKDF-SHA256 | info: `olcrtc-core/v1/<purpose>` |
| AEAD | XChaCha20-Poly1305 | 24-байтный nonce, 16-байтный tag |
| Nonce | `crypto/rand` | Случайный на каждое сообщение |

### Forward Secrecy

Каждая сессия использует **эфемерные** X25519-ключи. При компрометации PSK злоумышленник не сможет расшифровать прошлые перехваченные сессии — эфемерные ключи не сохраняются нигде.

### Аутентификация сообщений

AEAD обеспечивает аутентификацию каждого фрейма. Любая модификация или воспроизведение приведут к ошибке и разрыву соединения.

## Рекомендации по развёртыванию

### Генерация и хранение ключей

```bash
# Генерация
openssl rand -hex 32 > /etc/olcrtc/key.hex
chmod 600 /etc/olcrtc/key.hex
chown olcrtc:olcrtc /etc/olcrtc/key.hex

# Использование из файла
OLCRTC_KEY=$(cat /etc/olcrtc/key.hex) olcrtc-core -mode server ...
```

**Правила:**
- Храните ключ в файле с правами `600`, владелец — пользователь сервиса.
- Не помещайте ключ в Docker-образ или git-репозиторий.
- Используйте секрет-менеджеры (HashiCorp Vault, AWS Secrets Manager) в production.
- Ротируйте ключ каждые 30–90 дней.

### Сетевая изоляция

- Сервер не требует входящих TCP-портов для olcrtc (P2P через WebRTC).
- Разрешите только: `443/tcp`, `80/tcp`, `3478/udp` (STUN).
- Для API-запросов к провайдеру через ненадёжные сети используйте `-socks`.

### Изоляция процесса (systemd)

```ini
[Service]
User=olcrtc
Group=olcrtc
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true
CapabilityBoundingSet=
AmbientCapabilities=
```

## Ограничения модели безопасности

| Угроза | Защищает? | Примечание |
|--------|:---------:|-----------|
| Перехват трафика | ✅ | XChaCha20-Poly1305 + forward secrecy |
| Модификация трафика | ✅ | AEAD-аутентификация каждого фрейма |
| Повторное воспроизведение | ✅ | Случайный nonce + AEAD |
| Компрометация PSK → прошлые сессии | ✅ | Эфемерные X25519-ключи |
| Анонимность от провайдера | ❌ | Провайдер видит ваш IP |
| Сокрытие факта использования сервиса | ⚠️ | Трафик выглядит как WebRTC-звонок |
| Компрометация устройства | ❌ | Выходит за рамки ядра |
